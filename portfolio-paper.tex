    \section{WashFit | 부트캠프 프로젝트}

    \subsection{프로젝트 배경 및 목표}

    기존에 사용자들은 세차 용품 정보나 리뷰를 찾아보기 위해 특정 온라인 카페(커뮤니티)에 가입해야 하는 번거로움을 겪었습니다. 또한, 어떤 정보를 신뢰해야 할지 판단하기 어렵게 만드는 바이럴 마케팅과 수많은 유튜브 영상에 대한 피로감도 문제점으로 존재했습니다. \\ \\ 이처럼 세차 관련 정보가 여러 곳에 흩어져 있고 신뢰성을 판단하기 어려워, 사용자들이 필요로 하는 정보를 종합적으로 제공할 플랫폼의 필요성을 발견하였습니다. \\

    \begin{itemize}
        \item 안전한 제품 선택 지원: 세차 용품의 성분 및 안전 정보를 제공하여 사용자가 안전하게 제품을 선택할 수 있도록 돕습니다.
        \item 통합 정보 제공: 사용자들이 원하는 세차 용품 정보와 리뷰를 제공하고, 자신의 차량에 맞는 제품을 추천받을 수 있도록 합니다.
        \item 지역 기반 정보 제공: 사용자가 거주하는 동네의 좋은 세차장 정보를 추천하고 리뷰를 등록할 수 있는 기능을 제공합니다.
    \end{itemize}

    \subsection{문제 및 해결방법}

    \subsubsection{Spring Batch를 활용한 데이터 수집 및 가공 자동화}

    \needspace{8\baselineskip}

    \paragraph{문제 상황}

    \needspace{5\baselineskip}

    \paragraph{핵심 데이터의 외부 의존성} \par
    WashFit 서비스의 가장 중요한 기능은 사용자에게 '세차용품 안전 정보'를 제공하는 것입니다. 이 데이터는 서비스의 핵심 가치와 직결되지만, 자체적으로 생산하는 데이터가 아니라 '생활환경안전정보시스템(초록누리)'이라는 외부 공공 데이터 포털에서 API를 통해 가져와야 했습니다. 이처럼 서비스의 핵심 기능이 외부에 완전히 의존하는 구조는 데이터 관리의 어려움을 야기했습니다.

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{데이터 관리의 주요 과제} \par
    외부 데이터에 의존함에 따라 두 가지 주요 문제가 발생했습니다.
    \begin{itemize}
        \item \textbf{최신성(Freshness) 문제:} 공공 데이터 포털의 정보는 언제든지 업데이트될 수 있습니다. 만약 WashFit의 데이터가 제때 갱신되지 않으면, 사용자에게 잘못되거나 오래된 정보를 제공하게 될 위험이 있었습니다.
        \item \textbf{일관성(Consistency) 문제:} 서비스의 데이터베이스와 외부 데이터 원본 간의 정합성을 유지해야 했습니다. 이 과정을 수동으로 진행할 경우, 실수가 발생하거나 누락되는 데이터가 생길 수 있었습니다.
    \end{itemize}

    \par이러한 문제들 때문에, 사람의 개입 없이도 데이터를 주기적으로 동기화하여 최신성과 일관성을 자동으로 보장할 수 있는 안정적인 방안이 반드시 필요했습니다.\\


    \needspace{8\baselineskip}

    \paragraph{해결 방안}

    이 문제를 해결하기 위해 \textbf{Spring Batch}를 도입하여 안정적인 \textbf{ETL(Extract, Transform, Load) 파이프라인}을 구현했습니다. \\

    % % --- Figure 삽입 ---
%     \begin{center}
%         \includegraphics[width=0.7\linewidth]{images/spring-batch-architecture}
%         \captionof{figure}{Spring Batch 데이터 처리 파이프라인}
%         \label{fig:spring-batch}
%     \end{center}
% % --- Figure 삽입 ---

    \needspace{5\baselineskip}

    \paragraph{1단계: 외부 데이터 Staging} \par
    첫 번째 단계는 외부 데이터를 서비스 내의 임시 공간으로 가져오는 과정입니다.
    Spring Batch의 \texttt{ItemReader}를 사용하여 '초록누리' 외부 공공 API에서 제공하는 제품 안전 정보를 청크(Chunk) 단위로 읽어왔습니다. 이렇게 읽어온 원본 데이터는 즉시 가공하지 않고, 그대로 임시 테이블인 `rawData` 테이블에 저장(Staging)했습니다. 이 방식을 통해 외부 시스템과의 통신 부하를 최소화하고 원본 데이터를 먼저 확보하는 데 집중했습니다. \\

    \needspace{5\baselineskip}

    \paragraph{2단계: 데이터 가공 및 서비스 적재} \par
    두 번째 단계는 Staging된 데이터를 실제 서비스에서 사용할 형태로 가공하여 최종 목적지에 저장하는 과정입니다.
    `rawData` 테이블에 데이터 저장이 완료되면, 데이터베이스 트리거가 발동하여 다음 배치 작업을 실행시켰습니다. 이 작업에서는 \texttt{ItemProcessor}가 `rawData` 테이블에 쌓인 데이터를 검사하고 WashFit 서비스에 필요한 형태로 정제 및 가공했습니다. 가공이 완료된 데이터는 \texttt{ItemWriter}를 통해 최종적으로 사용자가 직접 조회하게 될 서비스 테이블에 저장(Load)되었습니다.

    \vspace{1em}
    \needspace{5\baselineskip}

    \paragraph{3단계: 실행 및 관리 자동화} \par
    전체 파이프라인의 실행과 관리는 Jenkins를 통해 자동화했습니다. 배치 모듈을 독립적인 .jar 파일로 패키징하고 이를 Jenkins 가 조작할 수 있도록 전달하였습니다. 그 뒤, 스케줄링 기능을 이용하여(예: 일요일 새벽 1시) 실행할 작업의 Spring Bean 이름울 통해 배치 작업을 트리거해 실행되도록 하였습니다. 그 이후, Jenkins 대시보드를 통해 배치 작업의 성공 여부를 시각적으로 관리할 수 있었습니다.\\

    \needspace{5\baselineskip}

    \paragraph{구현 결과}

    구현된 배치 작업이 매주 정해진 시간에 자동으로 실행되도록 설정하여, 별도의 수동 개입 없이도 사용자에게 항상 최신 데이터를 제공할 수 있는 기반을 마련했습니다. 이를 통해 운영 리소스를 절감하고 서비스에 데이터를 자동화하여 제공할 수 있었습니다.

    \subsubsection{API 응답 성능 병목 구간 발견 및 응답 속도 개선}

    \needspace{8\baselineskip}

    \paragraph{문제 상황}

    \needspace{5\baselineskip}

    \paragraph{1단계: 문제 발견} \par
    \textbf{Grafana}와 \textbf{Prometheus} 모니터링 시스템을 통해 `전체 제품 목록 조회 API`의 응답 속도가 저하되는 성능 병목 현상이 발견되었습니다. 특히 서버 배포 직후 초기 조회 속도가 평균 \textbf{748ms}에 달하는 등 지연이 심각한 것으로 확인되었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{2단계: 근본 원인 분석} \par
    팀원과 함께 원인을 분석한 결과, 데이터 페이징 처리에 사용된 \textbf{`Pageable` 객체}가 근본적인 원인이었음을 확인했습니다. \texttt{Pageable} 객체는 페이지네이션을 위해 내부적으로 데이터의 총 개수를 파악하는 불필요한 \textbf{`count` 쿼리}를 추가로 실행하였고, 이 쿼리가 API의 응답 성능을 저하시키는 주된 병목점이었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{3단계: 불필요한 로직 확인} \par
    UI/UX 측면에서 해당 API를 사용하는 화면은 총 페이지 수를 표시하지 않고 '무한 스크롤' 방식으로 다음 데이터 목록만 불러오는 기능으로 충분했습니다. 따라서, 백엔드에서 비용이 많이 드는 \texttt{count} 쿼리를 통해 전체 페이지 수를 계산하는 것은 실제 기능 요구사항에 부합하지 않는 불필요한 작업이었습니다.\\


    \needspace{8\baselineskip}

    \paragraph{해결 방안}

    \needspace{5\baselineskip}

    \paragraph{핵심 해결 방안: \texttt{Pageable}을 \texttt{Slice}로 교체} \par
    문제의 근본 원인이었던 불필요한 \texttt{count} 쿼리를 제거하기 위해, JPA 페이징 구현체를 교체하는 방안을 채택했습니다. UI/UX에서 \textbf{'무한 스크롤'} 기능을 사용하므로 전체 페이지 수가 필요 없다는 점에 착안하여, 기존의 \textbf{Pageable} 객체를 다음 페이지 존재 여부만 확인하는 \textbf{Slice} 객체로 교체를 제안하고 적용하였습니다. 이 변경으로 \texttt{count} 쿼리 실행을 제거하여 응답 속도를 개선했습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{코드 품질 개선: QueryDSL로 리팩토링} \par
    성능 개선 작업을 진행하는 과정에서 동료의 제안에 따라, 기존에 JPQL로 작성되었던 쿼리를 \textbf{QueryDSL}로 함께 리팩토링하였습니다. 이를 통해 성능 개선과 더불어, 컴파일 시점에 쿼리의 문법 오류를 발견할 수 있게 되어 코드의 안정성과 유지보수성을 향상시켰습니다.\\
    두 가지 해결 방안을 적용한 결과, 기존 평균 \textbf{748ms}였던 API 응답 속도를 \textbf{463ms}까지 단축시켜 총 \textbf{38\%의 성능 개선}을 달성했습니다.\\
    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{구현 결과}

    \paragraph{정량적 성과: 응답 속도 38\% 개선} \par
    \texttt{Slice} 적용과 QueryDSL 리팩토링을 통해 API 성능이 크게 향상되었습니다. 개선 작업을 통해, 기존 평균 748ms에 달하던 API 응답 속도를 최종적으로 463ms까지 단축시켰습니다. 이는 총 38%에 달하는 응답 속도 개선 성과입니다. 개선 과정은 다음과 같이 단계적으로 이루어졌습니다.\\

        \item 2단계 (\texttt{Slice} 적용): 545ms \(\rightarrow\) 463ms (추가 15\% 개선)

    성능 개선과 함께 진행된 QueryDSL로의 리팩토링은 코드 품질 측면에서도 긍정적인 결과를 가져왔습니다. 문자열 기반의 JPQL과 달리, QueryDSL은 자바 코드로 쿼리를 작성하므로 컴파일 시점에 문법 오류를 감지할 수 있게 되었습니다. 이를 통해 런타임 환경에서 발생할 수 있는 쿼리 관련 오류를 사전에 방지하여 서비스의 안정성과 코드의 유지보수성을 높였습니다.

    \paragraph{문제 상황}

    \needspace{5\baselineskip}

    \paragraph{1단계: 기능적 배경} \par
    문제 상황은 `아이디/비밀번호 찾기` 기능에서 시작되었습니다. 해당 기능은 사용자가 비밀번호 재설정을 요청할 경우, 이메일로 고유한 인증 링크를 발송하는 방식으로 동작했습니다.\\

    \needspace{5\baselineskip}

    \paragraph{2단계: 보안 취약점} \par
    핵심적인 보안 취약점은 이메일로 발송된 인증 링크가 시간제한 없이 계속 유효하다는 점이었습니다. 링크가 한번 생성되면 만료되지 않고 영구적으로 기능했기 때문에, 시간이 지나도 해당 링크를 통한 접근이 항상 허용되었습니다. 단순히 생각해보면 기존에 사용하던 데이터베이스 테이블에 만료일자 컬럼을 추가하여 사용할 수 있었지만, 단지 링크 만료시간만을 위해 운영중인 데이터베이스에 컬럼을 추가한다고 가정하고 생각해보니 이는 부담으로 다가왔습니다. (실제 사용자가 없어 데이터 양이 적었을 뿐, 실시간으로 서버를 운영중에 있었습니다.)\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{3단계: 위협 시나리오 및 영향} \par
    이러한 구조는 심각한 2차 피해로 이어질 수 있는 위협을 내포하고 있었습니다. 만약 사용자의 이메일 계정이 탈취될 경우, 공격자는 과거에 수신된 비밀번호 재설정 이메일을 찾아내어 만료되지 않은 인증 링크를 악용할 수 있었습니다. 이를 통해 공격자는 최종적으로 WashFit 서비스 계정의 소유권까지 탈취할 수 있는 2차적인 보안 위협이 존재했습니다. 따라서 인증 요청의 유효 시간을 제어할 수 있는 명확한 수단이 반드시 필요했습니다.\\



    \needspace{8\baselineskip}

    \paragraph{해결 방안}

    \needspace{5\baselineskip}

    \paragraph{1단계: 기술 선택: Redis 도입} \par
    영구적인 인증 링크 문제를 해결하기 위해, 빠르고 효율적인 In-Memory 데이터 저장소인 Redis를 도입했습니다. Redis가 기본적으로 제공하는 데이터 유효 시간 설정 기능인 TTL(Time To Live)을 활용하는 것이 핵심이었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{2단계: 구현 로직} \par
    인증 요청의 유효 시간을 제어하기 위해 다음과 같은 2단계 로직을 구현했습니다.\\
    \begin{enumerate}
        \item 인증 토큰 생성 및 저장: 사용자가 비밀번호 초기화를 요청하면, 시스템은 임의의 고유한 인증 토큰을 생성합니다. 이 토큰은 사용자의 식별 정보와 함께 Redis에 저장되며, 이때 짧은 유효 시간(\texttt{TTL})이 함께 설정됩니다.
        \item 인증 토큰 검증: 사용자가 이메일로 받은 인증 링크를 클릭하여 접근하면, 시스템은 링크에 포함된 토큰 값을 Redis에서 조회합니다. Redis에 해당 토큰이 유효하게 존재하는 경우에만 인증을 통과시키고, \texttt{TTL}이 만료되어 토큰이 사라진 경우에는 요청을 거부합니다.
    \end{enumerate}

    \par

    \needspace{10\baselineskip}

    \paragraph{구현 결과}


    \needspace{5\baselineskip}

    \paragraph{인증 링크의 유효 기간 제어} \par
    Redis의 \texttt{TTL} 기능을 도입한 결과, 비밀번호 재설정을 위한 인증 링크가 단기간 동안만 유효하도록 성공적으로 구현했습니다. 이전처럼 링크가 영구적으로 유효한 것이 아니라, Redis에 설정된 \texttt{TTL} 시간이 지나면 자동으로 만료되어 더 이상 사용할 수 없게 되었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{보안 위협 차단} \par
    인증 링크가 짧은 시간 동안만 유효하기 때문에, 사용자의 이메일 계정이 해킹되거나 인증 링크가 외부에 유출되더라도 시간이 지나면 해당 링크가 무효화됩니다. 이를 통해 공격자가 과거의 이메일을 통해 서비스 계정의 접근 권한을 탈취하는 시나리오를 효과적으로 방지하여 아이디/비밀번호 찾기 서비스의 보안 수준을 향상시킬 수 있었습니다.

    \section{KernelEngine | 부트캠프 프로젝트}

    \subsection{프로젝트 배경 및 목표}

    KernelEngine은 기술 블로그 게시글을 주기적으로 수집하고 키워드 검색 기능을 제공하는 웹 서비스입니다. 2023년 11월부터 12월까지 3인 팀 프로젝트로 진행되었으며, 자동화된 데이터 수집과 효율적인 검색 시스템 구축에 중점을 두었습니다.


    \needspace{10\baselineskip}

    \subsection{사용 기술}

    \begin{itemize}
        \item \textbf{Backend:} Java (17), Spring Boot, Spring Data JPA, Spring Security
        \item \textbf{Database:} MySQL
        \item \textbf{Template Engine:} Thymeleaf
        \item \textbf{Search Engine:} Elasticsearch
        \item \textbf{Infrastructure:} AWS EC2
        \item \textbf{CI/CD:} Github Actions
    \end{itemize}

    \subsection{문제 및 해결방법}

    \subsubsection{상황에 맞는 검색 기술 도입 및 성능 비교 분석}

    \needspace{8\baselineskip}

    \paragraph{문제 상황}

    \paragraph{검색 기능의 단계적 한계 봉착} \par
    프로젝트 초기, 검색 기능 구현을 위해 MySQL의 \texttt{LIKE} 연산자를 사용했습니다. 하지만 \texttt{\%keyword\%} 형태의 쿼리는 인덱스를 활용하지 못해 데이터가 증가할수록 성능 저하가 발생했고, 단순히 키워드 포함 여부만 확인하여 검색 결과의 정확도가 떨어지는 문제를 발견했습니다.
    \paragraph{}
    이 문제를 해결하고자 MySQL FULLTEXT검색을 도입했습니다. 전문 검색 인덱스를 활용하여 LIKE 방식보다 일부 개선된 성능을 보였지만, n-gram 알고리즘의 토큰화 단위 설정에 따라 특정 검색어가 누락되는 등 검색 품질의 한계가 여전히 존재했습니다. 이처럼 RDBMS 기반 검색 기술만으로는 사용자의 복잡한 검색 의도를 만족시키기 어렵다고 판단하여, ElasticSearch 도입을 검토하게 되었습니다.\\

    \needspace{8\baselineskip}

    \paragraph{해결 방안}

    \paragraph{1단계: 기술 학습 및 성능 비교} \par
    \textbf{MySQL}과 전문 검색엔진 \textbf{Elasticsearch}의 장단점을 명확히 파악하기 위해 사전학습을 병행한 테스트를 진행했습니다. 약 1,000개의 게시글 데이터를 대상으로 여러 키워드에 대한 응답 속도를 측정하고, 검색 결과의 정확도를 정성적으로 비교 평가했습니다.

    \vspace{1em}
    \needspace{8\baselineskip}
    \paragraph{2단계: 기술별 장단점 분석 및 역할 정의} \par

    \begin{itemize}
        \item \textbf{MySQL \texttt{LIKE} 방식의 장점}
        \begin{itemize}
            \item 간단한 구현: 별도 인프라 없이 기존 JPA 환경에서 즉시 구현 가능
            \item 데이터 일관성: DB 트랜잭션과 동기화되어 실시간 데이터 정합성 보장
            \item 정렬 최적화: DB 인덱스를 활용한 최신순/조회수순 정렬에 유리
        \end{itemize}
        \vspace{0.5em}
        \needspace{5\baselineskip}
        \item \textbf{Elasticsearch 방식의 장점}
        \begin{itemize}
            \item 높은 검색 품질: 형태소 분석, 유사도 점수(Scoring), 동의어 처리 등 고품질 검색 제공
            \item 뛰어난 성능: 역 인덱스(Inverted Index) 구조로 대용량 텍스트 검색에 최적화
        \end{itemize}
    \end{itemize}


    \needspace{8\baselineskip}

    \paragraph{구현 결과}

    \paragraph{요구사항에 최적화된 하이브리드 검색 전략 구현} \par
    PoC와 기술 분석 결과를 바탕으로, 각 기능의 요구사항에 맞는 기술을 선택적으로 적용하는  전략을 최종 구현했습니다.

    \vspace{1em}
    \paragraph{역할 분담 전략}
    \begin{itemize}
        \item MySQL \texttt{LIKE} \(\rightarrow\) 정렬 중심 검색: 최신순, 조회수순 정렬이 핵심적인 검색 요구사항일 경우 MySQL을 사용했습니다.

        \item Elasticsearch \(\rightarrow\) 정확도 중심 검색: 게시글의 제목과 본문을 아우르는 핵심 콘텐츠 검색 기능에는 Elasticsearch를 도입했습니다. 복잡한 텍스트 분석과 연관도 점수 기반의 정렬을 통해 사용자에게 훨씬 더 정확하고 만족도 높은 검색 결과를 제공했습니다.
    \end{itemize}

    \needspace{10\baselineskip}
    \subsubsection{Github Actions를 활용한 CI 파이프라인 구축}

    \needspace{8\baselineskip}

    \paragraph{문제 상황}

    \paragraph{코드 통합 프로세스의 비효율성} \par
    CI(Continuous Integration) 파이프라인이 부재한 상황에서는, 새로운 코드를 \texttt{main} 브랜치에 통합하기 전 코드의 정합성을 검증하는 모든 과정이 개발자에게 수동으로 의존하고 있었습니다. 이로 인해 두 가지 주요한 문제가 발생했습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{1. 코드 베이스의 안정성 저하 위험} \par
    빌드 실패나 테스트를 통과하지 못하는 코드가 \texttt{main} 브랜치에 Merge될 수 있는 잠재적인 위험이 항상 존재했습니다. 각기 다른 개발 환경으로 인해 문제가 발견되지 않은 코드가 그대로 통합될 경우, 팀 전체의 작업을 중단시키는 문제로 이어질 수 있습니다. 이는 항상 실행 가능한 상태의 코드 베이스를 유지하는 데 큰 걸림돌이 되었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{2. 비효율적인 코드 리뷰와 생산성 저하} \par
    동료의 코드를 리뷰하는 리뷰어는 코드의 논리적인 부분을 검토하기에 앞서, 해당 코드가 잘 실행되는지 부터 의심해야 했습니다. 이러한 반복적인 검증 작업은 생산성을 저해하는 요인이 되었습니다.\\


    \needspace{8\baselineskip}

    \paragraph{해결 방안}


    \needspace{5\baselineskip}

    \paragraph{1단계: 기술 선택 및 설정} \par
    수동 검증의 비효율성과 코드 안정성 저하 문제를 해결하기 위해, GitHub에 내장된 자동화 도구인 Github Actions를 활용하여 CI(Continuous Integration) 파이프라인을 구축했습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{2단계: 워크플로우 자동 실행 (Triggers)} \par
    작성된 워크플로우는 다음과 같은 두 가지 주요 이벤트가 발생할 때마다 자동으로 실행되도록 설정했습니다.
    \begin{itemize}
        \item 새로운 코드 변경 사항을 제안하는 \texttt{Pull Request}가 생성될 때
        \item 코드 변경 사항이 최종적으로 \texttt{main} 브랜치에 Push될 때
    \end{itemize}

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{3단계: 자동화된 검증 절차 (Jobs)} \par
    워크플로우가 실행되면, 다음과 같은 검증 절차가 자동으로 수행됩니다.
    \begin{enumerate}
        \item 소스 코드 체크아웃: 변경 사항이 적용된 소스 코드를 자동으로 내려받습니다.
        \item 빌드 및 테스트 실행: Gradle Wrapper를 사용하여 \texttt{./gradlew build}와 \texttt{./gradlew test} 명령어를 순차적으로 실행합니다. 이를 통해 프로젝트가 오류 없이 빌드되고, 모든 단위 테스트를 통과하는지 검증합니다.
    \end{enumerate}
    \\이 과정 중 하나라도 실패하면 워크플로우는 실패로 기록되고, Pull Request에 실패 상태가 표시되어 Merge가 방지됩니다.\\



    \needspace{5\baselineskip}

    \paragraph{구현 결과}

    \paragraph{ 각 팀원이 서로 코드 리뷰를 남길 때, 빌드하고 테스트하는 반복적인 검증 작업을 CI에 위임하고 오직 코드의 로직과 구조 등에만 집중하여 리뷰를 진행할 수 있었습니다.}

    \needspace{10\baselineskip}
    \subsubsection{Spring Scheduler를 활용한 주기적인 데이터 수집 및 통계 생성}

    \needspace{8\baselineskip}

    \paragraph{문제 상황}


    \needspace{5\baselineskip}

    \paragraph{반복적인 데이터 처리 작업의 자동화 필요성} \par
    KernelEngine 서비스 운영을 위해, 주기적으로 실행되어야 하는 두 가지 주요한 수동 작업을 자동화할 필요성이 있었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{1. 콘텐츠 주기적 확보의 어려움} \par
    서비스의 핵심 가치는 사용자에게 여러 블로그의 '게시글'을 제공하는 것이었습니다. 특정 게시글이 발행되었는지 아닌지를 직접 지켜보고 있다가 콘텐츠를 등록하는 것은 너무나 비효율적이기 때문에, 서비스에 등록된 블로그에 새로 게시되는 글들을 지속적으로 확인하고 데이터베이스로 가져오는 작업이 필수적이었습니다. 이 작업을 수동으로 실행할 경우, 데이터가 즉시 반영되지 않아 정보가 낡은 상태로 방치될 수 있습니다. 따라서 서비스의 핵심 요구 사항을 지키기 위해 신규 게시글을 주기적으로 수집할 안정적인 방안이 필요했습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{2. 수동 통계 집계의 비효율성} \par
    `일간 회원 가입자 수`와 같은 통계 지표를 집계해보고자 시도 하였습니다. 다만 매일 담당 팀원이 직접 쿼리를 실행하여 집계하는 것은 매우 번거롭고 비효율적인 과정이었습니다. 따라서 이러한 반복적인 통계 생성 작업을 자동으로 집계할 필요가 있었습니다.\\



    \needspace{8\baselineskip}

    \paragraph{해결 방안}


    \needspace{5\baselineskip}

    \paragraph{기술 선택: Spring Scheduler} \par
    두 가지 반복적인 수동 작업을 자동화하기 위해, Spring Framework가 기본적으로 제공하는 Spring Scheduler를 해결 방안으로 채택했습니다. 외부 스케줄링 도구 없이 애플리케이션 내에서 간단한 어노테이션과 설정만으로 주기적인 작업을 관리할 수 있다는 장점이 있었습니다.\\

    \vspace{1em}

    \needspace{5\baselineskip}

    \paragraph{구현 내용} \par
    요구사항을 해결하기 위해, cron 표현식을 이용하여 실행 주기를 정의하는 두 개의 독립적인 스케줄링 작업을 구현했습니다.\\


    \needspace{10\baselineskip}
    \begin{itemize}
        \item 블로그 크롤링 작업: 등록된 블로그의 신규 게시글을 수집하는 크롤링 작업은 매일 특정 시간에 주기적으로 실행되도록 스케줄을 설정했습니다.
        \item 일간 통계 집계 작업: `일간 가입자 수`를 집계하는 통계 작업은 하루의 데이터를 온전히 마감하기 위해 매일 자정에 실행되도록 스케줄을 설정했습니다.
    \end{itemize}

    \needspace{5\baselineskip}
    \begin{lstlisting}[language=Java, caption=블로그 게시글 수집 작업 예시]
@Scheduled(cron = "0 0 3 * * ?")
public void runAddFeedFromDataLoaderJob() {
    ...
    try {
        jobLauncher.run(addFeedFromBlogJobConfig.addFeedFromDataLoaderJob(), jobParameters);
    } catch (...)
}
    \end{lstlisting}

    \needspace{8\baselineskip}

    \paragraph{구현 결과}


    \needspace{5\baselineskip}

    \paragraph{콘텐츠 데이터 확보 일부 자동화} \par
    매일 특정 시간에 크롤링 작업이 자동으로 실행됨으로써, 수동 개입 없이도 콘텐츠 데이터의 확보를 보장할 수 있게 되었습니다.\\
