## 문제 상황 — 외부 API 데이터의 안정적 수집과 가공

WashFit의 핵심 데이터는 공공 데이터 포털(초록누리)에서 API로 가져오는 세차 용품 안전 정보입니다. API가 반환하는 신고 제품 총 건수는 약 37만 건이며, 페이지당 20건씩 응답합니다. 이 중 서비스에 필요한 세차 용품은 약 2만 건으로, 수집 후 필터링·가공이 필요합니다. 전체를 한 번에 조회하면 메모리 부담과 API 호출 실패 시 전체 재시작 문제가 있었습니다.

## 해결 방안 — Spring Batch 기반 2단계 ETL 파이프라인

Spring Batch를 도입하여 수집과 가공을 분리한 파이프라인을 구축했습니다. 분리한 이유는 두 가지입니다. 외부 API 데이터에 스키마가 깨지거나 내용이 잘못된 경우가 있어 수집 단계에서 먼저 실패시켜 격리하고, 서비스에 필요한 세차 용품만 별도로 가공하기 위해서입니다.

**1단계(Extract)**: 외부 API에서 페이지 단위로 데이터를 읽어 중간 테이블(reported_product, concerned_product, violated_product)에 적재합니다. Chunk Size는 10으로 설정했는데, 100건 단위로 처리하면 실패 시 원인이 되는 레코드를 특정하기 어렵고, 1건 단위는 처리 속도가 느려 절충한 값입니다.

**2단계(Transform & Load)**: 중간 테이블의 데이터를 서비스용 product 테이블로 변환·적재합니다. 37만 건 중 세차와 무관한 키워드(세탁, 방향제 등)를 필터링하여 약 2만 건을 최종 적재합니다.

## 구현 결과

- 수집·가공 분리로 외부 API의 데이터 오류를 수집 단계에서 격리, 서비스 테이블에 영향 없이 처리
- 10개 배치 Job을 제품 유형별(신고·주의·위해)로 독립 구성, `job.name` 파라미터로 선택 실행
