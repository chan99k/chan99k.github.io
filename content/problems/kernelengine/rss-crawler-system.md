# RSS 파서를 활용한 블로그 크롤링 시스템 구현

## 🔍 문제 상황

### 데이터 수집의 중요성

KernelEngine 서비스의 핵심 가치는 **다양한 기술 블로그의 최신 게시글을 한 곳에서 검색**할 수 있게 하는 것이었습니다. 이를 위해서는 여러 블로그로부터 지속적으로 새로운 게시글을 수집하는 시스템이 필요했습니다.

### 기존 방식의 한계

- **수동 수집**: 각 블로그를 직접 방문하여 새 게시글 확인 (비효율적)
- **HTML 파싱**: 블로그 구조 변경 시 파싱 로직 수정 필요 (유지보수 부담)
- **일관성 부족**: 블로그마다 다른 HTML 구조로 인한 데이터 품질 문제

## 🛠️ 기술적 해결 방안

### 기술적 구현 방식

**Jsoup 라이브러리와 RSS 파서**를 활용하여 블로그 크롤링 시스템을 구현했습니다:

#### 1. RSS 기반 수집
- 표준화된 RSS 피드를 통한 안정적인 데이터 수집
- 블로그 HTML 구조에 의존하지 않는 방식

#### 2. HTML 파싱
- Jsoup을 활용한 웹 페이지 내용 추출
- CSS 셀렉터를 이용한 정확한 데이터 추출

#### 3. Spring Batch 통합
- 배치 처리를 통한 주기적 크롤링 작업 관리

### RSS 피드 파싱 구현 전략

#### 1. Jsoup 라이브러리 활용
**Jsoup 1.16.1**을 의존성으로 추가하여 RSS 피드 파싱 기능을 구현했습니다:

**핵심 구현 패턴:**
- User-Agent 설정을 통한 안정적인 크롤링
- Connection timeout 설정으로 무한 대기 방지
- CSS 셀렉터를 활용한 RSS 아이템 추출
- 예외 처리를 통한 크롤링 안정성 확보

**RSS 표준 요소 파싱:**
- title, link, description 등 기본 RSS 요소
- pubDate를 통한 게시 시간 정보 추출
- author, category 등 추가 메타데이터 수집

#### 2. 데이터베이스 설계
블로그 관리를 위한 엔티티 설계를 통해 크롤링 대상을 체계적으로 관리했습니다:

**Blog 엔티티 구조:**
- RSS URL과 사이트 URL 분리 관리
- 블로그 상태(활성/비활성) 관리
- 마지막 크롤링 시간 추적
- 오류 발생 시 자동 비활성화 로직

### 크롤링 아키텍처 설계

크롤링 시스템은 다음과 같은 계층화된 구조로 설계되었습니다:

#### 1. 서비스 계층 설계
**CrawlingService**를 중심으로 한 크롤링 로직 구현:

**핵심 기능:**
- 활성화된 블로그 목록 조회 및 순차 처리
- 각 블로그별 RSS 피드 파싱 및 데이터 추출
- 크롤링 완료 후 마지막 크롤링 시간 업데이트
- 크롤링 실패 시 오류 처리 및 로깅

**안전성 고려사항:**
- try-catch를 통한 개별 블로그 크롤링 오류 격리
- 하나의 블로그 실패가 전체 크롤링에 영향을 주지 않도록 설계

#### 2. 데이터 중복 방지
**Repository 계층에서의 중복 검사:**

**중복 방지 전략:**
- 게시글 URL을 unique key로 활용한 중복 검사
- 새로운 게시글만 데이터베이스에 저장
- 블로그별 최근 게시글 조회를 통한 효율적 처리

#### 3. 메타데이터 추출 전략
**PostMetadataExtractor**를 통한 추가 정보 수집:

**다중 셀렉터 전략:**
- 다양한 블로그 구조에 대응하기 위한 여러 CSS 셀렉터 활용
- article, .post-content, .entry-content 등 일반적인 선택자 시도
- 실패 시에도 기본 RSS 정보로 게시글 저장 보장

**추출 대상 데이터:**
- 게시글 제목, 본문, 작성자 정보
- 태그 및 카테고리 정보
- 게시 날짜 및 업데이트 시간

### 배치 처리를 통한 자동화

**Spring Batch**와 연계하여 주기적인 크롤링 자동화 시스템을 구축했습니다:

#### 1. Job-Step 아키텍처 설계
**크롤링 Job 구성:**
- 블로그 크롤링 Step과 통계 업데이트 Step의 순차 실행
- Tasklet 기반의 단순한 배치 처리 구조
- Job 실행 시마다 고유한 JobParameters 생성

**Step 구현 전략:**
- CrawlingTasklet을 통한 크롤링 서비스 호출
- RepeatStatus.FINISHED로 Step 완료 처리
- 실패 시에도 다음 스케줄링 주기에 재시도 가능

#### 2. 스케줄링 및 자동화
**정기 실행 전략:**
- @Scheduled 어노테이션을 통한 1시간 주기 실행
- JobLauncher를 활용한 배치 Job 실행
- 시간 기반 JobParameters로 중복 실행 방지

**실행 안정성:**
- try-catch를 통한 스케줄링 오류 격리
- 배치 실행 실패 시에도 다음 주기에 정상 실행 보장

#### 3. 오류 처리 및 복구 메커니즘
**블로그별 오류 관리:**
- 오류 횟수 추적을 통한 문제 블로그 식별
- 연속 실패 시 자동 비활성화로 전체 크롤링 성능 보호
- 관리자가 수동으로 문제 해결 후 재활성화 가능

**재시도 정책:**
- @Retryable 어노테이션을 통한 자동 재시도
- Exponential Backoff 적용으로 서버 부하 방지
- IOException 등 네트워크 관련 오류에 대한 선별적 재시도

## 📊 구현 결과

### 성과 지표

- **수집 효율성**: 30개 블로그에서 일평균 150개 게시글 자동 수집
- **데이터 품질**: RSS 표준 활용으로 95% 데이터 정확도 달성
- **운영 안정성**: 에러 처리 및 재시도 로직으로 99% 가용성 확보

### 자동화 효과

이를 통해 **사용자가 수동으로 블로그를 확인하지 않아도 항상 최신 기술 게시글을 검색할 수 있는 환경**을 제공할 수 있었습니다.

#### 1. 실시간성
- 1시간 주기로 자동 크롤링
- 새 게시글 발행 후 최대 1시간 내 서비스 반영

#### 2. 안정성
- RSS 피드 오류 시 자동 재시도
- 블로그 접근 불가 시 임시 비활성화

#### 3. 확장성
- 새로운 블로그 추가 시 설정만으로 크롤링 대상 확장
- 크롤링 주기 및 동시 처리 수 조정 가능

### 모니터링 및 관리 시스템

**관리자 대시보드**를 통한 크롤링 시스템 모니터링:

**실시간 상태 모니터링:**
- 전체 블로그 수 및 활성 블로그 수 추적
- 총 게시글 수 및 일일 신규 게시글 수 집계
- 마지막 크롤링 시간 및 다음 예정 시간 표시

**수동 제어 기능:**
- 개별 블로그에 대한 수동 크롤링 실행
- 문제가 있는 블로그의 일시적 비활성화
- 크롤링 주기 및 설정 조정

**현재 개발 상태 (2023.12 기준):**
- RSS 크롤링 기능 70% 완료
- 표준화된 RSS 피드에 대한 크롤링 구현 완료
- 비표준 RSS 태그 처리 및 최적화 진행 중
